# SQL Processor 학습 Checklist

이 문서는 이 프로젝트를 14일 동안 직접 이해하기 위한 체크리스트다.  
목표는 "코드를 읽었다"가 아니라 "직접 실행하고, 추적하고, 설명할 수 있다" 상태에 도달하는 것이다.

## 공통 학습 원칙

- [ ] 매일 코드 읽기만 하지 말고 반드시 직접 실행한다.
- [ ] 읽은 함수는 "입력 / 처리 / 출력 / 실패 조건" 4칸으로 정리한다.
- [ ] 매일 최소 10분은 손으로 흐름도나 표를 직접 작성한다.
- [ ] 모르는 C 문법은 이 프로젝트 밖의 작은 예제로 따로 확인한다.
- [ ] 매일 학습 끝에 "오늘 이해한 것 3개 / 아직 헷갈리는 것 2개 / 내일 확인할 것 1개"를 기록한다.

## Day 1. 프로젝트를 직접 실행해서 감 잡기

### 목표

- [ ] 프로그램이 실제로 어떻게 동작하는지 눈으로 확인한다.
- [ ] SQL 파일 입력, CLI 출력, CSV 파일 변화의 관계를 관찰한다.

### 실행 체크리스트

- [ ] 프로젝트 루트에서 현재 파일 구조를 확인한다.
- [ ] `Makefile`, `examples`, `data` 디렉터리의 역할을 대략 파악한다.
- [ ] `make`를 실행한다.
- [ ] `make test`를 실행한다.
- [ ] `cat examples/insert_user.sql`을 실행한다.
- [ ] `cat examples/select_users.sql`을 실행한다.
- [ ] `cat data/users.csv`를 실행해서 초기 상태를 본다.
- [ ] `./sql_processor examples/insert_user.sql`을 실행한다.
- [ ] 다시 `cat data/users.csv`를 실행해서 변경된 상태를 본다.
- [ ] `./sql_processor examples/select_users.sql`을 실행한다.

### 관찰 체크리스트

- [ ] `make`가 어떤 파일을 컴파일하는지 기록한다.
- [ ] `make test`가 별도 테스트 바이너리를 실행하는 구조인지 확인한다.
- [ ] INSERT 실행 전후 `data/users.csv`가 어떻게 달라졌는지 기록한다.
- [ ] SELECT 실행 시 헤더, 데이터 행, `Rows: N`이 어떤 순서로 출력되는지 기록한다.

### 기록 체크리스트

- [ ] 입력 파일 내용을 정리했다.
- [ ] 터미널 출력 결과를 정리했다.
- [ ] CSV 파일 변경 전/후를 정리했다.
- [ ] 전체 흐름을 3줄로 요약했다.

### 자기 점검

- [ ] 이 프로그램은 SQL 문자열이 아니라 SQL 파일을 입력으로 받는다고 설명할 수 있다.
- [ ] 데이터가 메모리가 아니라 파일에 저장된다고 설명할 수 있다.
- [ ] INSERT와 SELECT가 각각 어떤 파일을 읽고 쓰는지 설명할 수 있다.

### 완료 기준

- [ ] INSERT 실행 후 CSV에 새 줄이 생기는 것을 직접 확인했다.
- [ ] SELECT 실행 후 CSV 전체가 출력되는 것을 직접 확인했다.

## Day 2. 공용 자료구조 이해하기

### 목표

- [ ] `Statement` 구조체가 이 프로젝트의 핵심 내부 표현이라는 점을 이해한다.
- [ ] "파싱 결과를 구조체 하나로 표현한다"를 자기 말로 설명할 수 있다.

### 읽기 체크리스트

- [ ] [common.h](./common.h)를 처음부터 끝까지 읽었다.
- [ ] `MAX_SQL_LENGTH` 의미를 정리했다.
- [ ] `MAX_TABLE_NAME` 의미를 정리했다.
- [ ] `MAX_VALUES` 의미를 정리했다.
- [ ] `MAX_VALUE_LENGTH` 의미를 정리했다.
- [ ] `StatementType` 의미를 정리했다.
- [ ] `Statement` 각 필드 의미를 정리했다.

### 실습 체크리스트

- [ ] 왜 `StatementType`이 `INSERT`, `SELECT` 두 종류만 가지는지 적었다.
- [ ] `Statement` 각 필드 의미를 한 줄씩 적었다.
- [ ] 아래 SQL 2개가 각각 어떤 `Statement`로 바뀌는지 손으로 적었다.

```sql
INSERT INTO users VALUES (1, 'Alice', 20);
SELECT * FROM users;
```

### 작성 체크리스트

- [ ] 아래 표를 직접 채웠다.

| SQL | type | table_name | value_count | values |
| --- | --- | --- | --- | --- |
| INSERT INTO users VALUES (1, 'Alice', 20); |  |  |  |  |
| SELECT * FROM users; |  |  |  |  |

- [ ] 왜 파싱 결과를 문자열 그대로 두지 않고 구조체로 바꾸는지 적었다.
- [ ] 왜 `SELECT`에서는 `value_count`가 0이어도 되는지 적었다.
- [ ] 고정 길이 배열 방식의 장단점을 적었다.

### 자기 점검

- [ ] `Statement`만 보고 INSERT인지 SELECT인지 구분할 수 있다.
- [ ] `table_name`이 언제 채워지는지 설명할 수 있다.
- [ ] `values`가 언제 채워지는지 설명할 수 있다.
- [ ] 고정 길이 버퍼 방식이 학습용 MVP에 왜 적절한지 설명할 수 있다.

### 완료 기준

- [ ] `Statement` 구조체를 보지 않고도 주요 필드를 설명할 수 있다.
- [ ] SQL 한 줄이 `Statement`로 바뀌는 과정을 말로 설명할 수 있다.
