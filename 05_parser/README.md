# 05_parser

이 폴더의 목표는 **파서가 어떤 일을 담당하는지 먼저 선언하는 것**입니다.

지금 만드는 `parser.h`는 실제 파싱 코드를 다 쓰는 파일이 아닙니다.
대신, 파서가 밖으로 어떤 함수들을 제공할지 먼저 약속하는 파일입니다.

## 이 단계에서 만드는 파일

- `parser.h`

## 파서가 하는 일

파서는 SQL 문자열을 읽고,
그 문장이 `INSERT`인지 `SELECT`인지 구분한 뒤,
그 결과를 `Command` 구조체에 채워 넣습니다.

예를 들어 아래 SQL이 들어오면:

```sql
INSERT INTO materials VALUES ('LX2 선바이저', 'ASD', 'MMH', '21G');
```

파서는 대략 이런 정보를 만들어야 합니다.

- 명령 종류: `COMMAND_INSERT`
- 테이블 이름: `materials`
- 값 목록: `LX2 선바이저`, `ASD`, `MMH`, `21G`
- 값 개수: `4`

## `parser.h` 안의 함수들

### `init_command`

`Command` 구조체를 처음 사용할 때 기본값으로 초기화합니다.

예:

- type -> `COMMAND_UNKNOWN`
- table_name -> 비어 있음
- value_count -> 0

즉, 빈 주문서를 준비하는 단계입니다.

### `free_command`

지금 버전에서는 구조체 안에 동적 메모리를 직접 많이 쓰지 않더라도,
나중에 구조가 커질 수 있기 때문에 정리 함수 이름을 먼저 만들어 둡니다.

초보 단계에서는 "사용이 끝난 주문서를 정리하는 함수"라고 생각하면 충분합니다.

### `parse_sql`

이 함수가 핵심입니다.

역할:

- SQL 문자열을 읽는다.
- INSERT인지 SELECT인지 판단한다.
- `Command` 구조체를 채운다.
- 문제가 있으면 에러 메시지를 만든다.

즉, 문자열을 사람이 이해할 수 있는 구조로 바꾸는 입구입니다.

## 왜 헤더 파일에서 먼저 선언할까

`.h` 파일은 "이런 함수가 있다"고 알려 주는 약속서입니다.
`.c` 파일은 그 약속을 실제로 구현하는 곳입니다.

그래서 지금은 먼저 "파서 부서가 어떤 일을 맡는가"를 정하는 단계입니다.
